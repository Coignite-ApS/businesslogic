> **Predictable Business Intelligence вЂ” for AI and Humans**

This is the unified monorepo for the BusinessLogic platform. All services have been migrated from the legacy polyrepo into a modular, independently-scalable service mesh. The migration (iterations 00вЂ“05) is complete; historical plans are in `docs/migrations/` for reference.

## On Startup вЂ” Greet the User

When a new conversation starts, greet the user with a brief status overview. Do this by:

1. Read `docs/tasks/README.md` and scan task statuses across all services
2. Check `git branch --show-current` and `git log --oneline -3` for recent activity
3. Present a greeting like:

```

рџ“Љ Tasks: X completed | Y in-progress | Z planned
рџЊї Branch: <current branch> | Last: "<recent commit message>"

  /browser-qa                вЂ” verify UI changes in Chrome

4. If any tasks are `in-progress`, highlight them вЂ” the user may want to continue

## Agent Model Policy

**Opus** is required for roles that involve judgment, review, or strategic decisions.
**Sonnet** is used for implementation work where speed and cost matter.

## Key Architecture Rules

1. **Schema ownership**: Each service owns a PostgreSQL schema. Only the owner WRITES. Others may READ.
   - `cms.*` в†’ bl-cms
   - `ai.*` в†’ bl-ai-api
   - `formula.*` в†’ bl-formula-api
   - `flow.*` в†’ bl-flow
   - `gateway.*` в†’ bl-gateway

2. **Redis namespacing**: Each service prefixes keys: `gw:`, `ai:`, `fa:`, `fl:`, `cms:`. Shared rate-limit keys use `rl:`.

3. **Direct database access**: All services connect directly to PostgreSQL (correct at 5 services). No data service layer.

4. **Shared libraries** (`packages/`): Common queries (account lookup, rate limiting, caching) extracted to avoid duplication.

5. **Gateway for public traffic**: Client в†’ Cloudflare в†’ bl-gateway в†’ service. Internal traffic goes direct (no gateway).

6. **Directus is back-office only**: Directus handles admin UI, account management, and data CRUD. All compute (AI, formulas, flows) runs in dedicated services.

7. **Always use Makefile targets**: NEVER run raw `docker compose`, `npx directus-extension build`, or other infrastructure commands directly. Use `make <target>` from the project root (or `services/cms/` for CMS-specific tasks). This ensures consistent compose files, build flags, and snapshot naming conventions.

## Test-Driven Development (TDD)

**This project follows strict TDD. Every change must follow this cycle:**

1. **Write the test first** вЂ” define expected behavior before implementing
2. **Run the test вЂ” watch it fail** вЂ” confirm the test actually tests something
3. **Implement the minimum code** to make the test pass
4. **Run all tests** вЂ” ensure nothing else broke
5. **Refactor** вЂ” clean up while tests are green
6. **Commit** вЂ” hooks enforce tests pass before commit

### Test Frameworks by Service

| Service | Framework | Command | Config |
|---------|-----------|---------|--------|

| flow | Cargo test (Rust) | `cargo test --workspace` | Tests in each crate |
| cms extensions | Vitest + Node.js test runner | `npm test` (in extensions/) | `__tests__/` dirs |

### When to Write Tests

- **Extracting shared code to packages/:** Write tests for the package, then verify dependent services

## Claude Code Hooks

Hooks are configured in `.claude/settings.json` (committed to git, shared with team) and enforced automatically:

| Hook | Event | What it does |
|------|-------|-------------|
| `protect-branches.sh` | PreToolUse (Bash) | Blocks direct commits to main/master |
| `pre-commit-tests.sh` | PreToolUse (Bash) | Runs service-specific tests before git commit/push |
| `pre-commit-docs-check.sh` | PreToolUse (Bash) | Warns if service code changed but no task doc updated |
| `post-edit-lint.sh` | PostToolUse (Edit/Write) | Auto-formats files after edits (eslint, cargo fmt, gofmt) |
| `post-commit-task-eval.sh` | PostToolUse (Bash) | After git commit, evaluates in-progress tasks and prompts status update |

**How hooks work:**
- When Claude runs `git commit`, the PreToolUse hooks fire BEFORE the command executes
- `protect-branches.sh` checks current branch вЂ” blocks if on main
- `pre-commit-tests.sh` detects which services have staged changes, runs their tests
- If ANY test fails, the commit is blocked with exit code 2
- Claude must fix the failing tests before retrying the commit
- `pre-commit-docs-check.sh` warns when service code changes without a corresponding task doc update вЂ” ensures functionality changes are documented
- After a successful commit, `post-commit-task-eval.sh` checks in-progress tasks вЂ” if all Key Tasks checkboxes are checked, it tells Claude to mark the task as completed

**Hook files:** `.claude/hooks/` вЂ” all scripts must be executable (`chmod +x`)

## Database Snapshots

Rolling 5-copy backup in `infrastructure/db-snapshots/`:

```
