# BusinessLogic Platform — Monorepo

> **Predictable Business Intelligence — for AI and Humans**

This is the unified monorepo for the BusinessLogic platform. All services have been migrated from the legacy polyrepo into a modular, independently-scalable service mesh. The migration (iterations 00–05) is complete; historical plans are in `docs/migrations/` for reference.

## On Startup — Greet the User

When a new conversation starts, greet the user with a brief status overview. Do this by:

1. Read `docs/tasks/README.md` and scan task statuses across all services
2. Check `git branch --show-current` and `git log --oneline -3` for recent activity
3. Present a greeting like:

```
👋 Welcome to BusinessLogic.

📊 Tasks: X completed | Y in-progress | Z planned
🌿 Branch: <current branch> | Last: "<recent commit message>"

🔧 What you can do:
  /tasks                     — see full backlog, pick next task
  /tasks <service>           — filter by service (cms, ai-api, formula-api, flow, gateway)
  /robo-task <path>          — run autonomous loop on a task
  /robo-sprint               — run multiple tasks in parallel (Agent Teams)
  /robo-sprint formula-api   — parallel tasks for specific services
  /cto-review                — technical review after changes
  /devops-review             — infrastructure review
  /frontend-designer         — UI design review or build
  /browser-qa                — verify UI changes in Chrome
  /bizdev-strategy           — market & pricing analysis
  /project-review            — full health check
  /test-all                  — run all tests
  /dev-up                    — start Docker dev environment

💡 To add a new task: /tasks add <service>
```

4. If any tasks are `in-progress`, highlight them — the user may want to continue
5. Keep it concise — 10-15 lines max, no walls of text

## Key Architecture Rules

1. **Schema ownership**: Each service owns a PostgreSQL schema. Only the owner WRITES. Others may READ.
   - `cms.*` → bl-cms
   - `ai.*` → bl-ai-api
   - `formula.*` → bl-formula-api
   - `flow.*` → bl-flow
   - `gateway.*` → bl-gateway

2. **Redis namespacing**: Each service prefixes keys: `gw:`, `ai:`, `fa:`, `fl:`, `cms:`. Shared rate-limit keys use `rl:`.

3. **Direct database access**: All services connect directly to PostgreSQL (correct at 5 services). No data service layer.

4. **Shared libraries** (`packages/`): Common queries (account lookup, rate limiting, caching) extracted to avoid duplication.

5. **Gateway for public traffic**: Client → Cloudflare → bl-gateway → service. Internal traffic goes direct (no gateway).

6. **Directus is back-office only**: Directus handles admin UI, account management, and data CRUD. All compute (AI, formulas, flows) runs in dedicated services.

## Git Rules

- **No Co-Authored-By or Claude mentions in commit messages.**
- **Conventional commits**: `feat(ai-api):`, `fix(cms):`, `chore(infra):` etc.
- **Protected main**: NEVER commit directly to main. All work goes through dev.
- **Branch hierarchy**: `main` ← `dev` ← `feat/XX-description`

### Branching Strategy

```
main (protected — production-ready, merged via PR only)
 └── dev (integration branch)
      └── feat/XX-description (feature work)
```

**Rules:**
1. Branch features from `dev`: `git checkout -b feat/XX-description dev`
2. Merge completed features into `dev`: `git checkout dev && git merge feat/XX-description`
3. PR from `dev` to `main` when verified and stable
4. Hooks enforce this — direct commits to `main` are BLOCKED

### Commit Workflow

```bash
# Before committing, hooks automatically:
# 1. Check you're NOT on main (blocks if so)
# 2. Detect which services changed
# 3. Run tests for those services
```

## Test-Driven Development (TDD)

**This project follows strict TDD. Every change must follow this cycle:**

1. **Write the test first** — define expected behavior before implementing
2. **Run the test — watch it fail** — confirm the test actually tests something
3. **Implement the minimum code** to make the test pass
4. **Run all tests** — ensure nothing else broke
5. **Refactor** — clean up while tests are green
6. **Commit** — hooks enforce tests pass before commit

- **Fixing a bug:** Write a test that reproduces the bug before fixing it
- **Extracting shared code to packages/:** Write tests for the package, then verify dependent services

## Claude Code Hooks

Hooks are configured in `.claude/settings.json` (committed to git, shared with team) and enforced automatically:

| Hook | Event | What it does |
|------|-------|-------------|
| `protect-branches.sh` | PreToolUse (Bash) | Blocks direct commits to main/master |
| `pre-commit-tests.sh` | PreToolUse (Bash) | Runs service-specific tests before git commit/push |
| `pre-commit-docs-check.sh` | PreToolUse (Bash) | Warns if service code changed but no task doc updated |
| `post-edit-lint.sh` | PostToolUse (Edit/Write) | Auto-formats files after edits (eslint, cargo fmt, gofmt) |
| `post-commit-task-eval.sh` | PostToolUse (Bash) | After git commit, evaluates in-progress tasks and prompts status update |

**How hooks work:**
- When Claude runs `git commit`, the PreToolUse hooks fire BEFORE the command executes
- `protect-branches.sh` checks current branch — blocks if on main
- `pre-commit-tests.sh` detects which services have staged changes, runs their tests
- If ANY test fails, the commit is blocked with exit code 2
- Claude must fix the failing tests before retrying the commit
- `pre-commit-docs-check.sh` warns when service code changes without a corresponding task doc update — ensures functionality changes are documented
- After a successful commit, `post-commit-task-eval.sh` checks in-progress tasks — if all Key Tasks checkboxes are checked, it tells Claude to mark the task as completed

**Hook files:** `.claude/hooks/` — all scripts must be executable (`chmod +x`)

## Task Completion Checklist

Before marking ANY task as done, verify:

1. **Tests written** — new behavior has test coverage
2. **Tests pass** — `./scripts/test-all.sh` (or service-specific)
3. **No regressions** — existing tests still pass
4. **Linted** — no lint errors in changed files
5. **Committed** — changes committed on the correct branch
6. **Contract tests** — if API changed, `./scripts/test-contracts.sh` passes
7. **Run `/project-review`** — after completing any iteration or significant task

## Project Review Skill

The `/project-review` skill performs a comprehensive health check of the entire project:

- **Tests**: Runs all test suites, reports pass/fail per service
- **Git**: Checks branch, uncommitted changes, recent history
- **Services**: Health checks all running Docker services
- **Database**: Takes a snapshot (rolling 5 copies), exports current schema
- **Documentation**: Verifies all required docs exist and are current
- **Next steps**: Suggests what to do next based on current state

**Usage:**
```bash
/project-review           # Full review
/project-review quick     # Skip database dump
/project-review report    # Generate markdown report in docs/reports/
```

**When to run:**
- After a significant code change
- When you want to understand where the project stands
- Before presenting status to stakeholders

## Database Snapshots
