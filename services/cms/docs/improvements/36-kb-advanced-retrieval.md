## Goal

Upgrade the KB retrieval pipeline from basic vector search to a **state-of-the-art hybrid retrieval system** — without adding new infrastructure. All techniques use existing PostgreSQL + pgvector + Redis stack.

This is the technical quality backbone. #34 (curated answers) gives admins manual control; #35 (feedback learning) adds learning over time; this improvement makes the **underlying retrieval fundamentally better**.

## Why Not Neo4j?

Researched thoroughly. **GraphRAG is not the right move:**

| Factor | Finding |
|--------|---------|
| **Simple factual queries** (80%+ of business KB) | GraphRAG is **13.4% less accurate** than vector RAG |
| **Cost** | 3-5x more expensive; Microsoft GraphRAG uses 610K tokens per retrieval vs ~100 |
| **Latency** | 2.3x higher average |
| **Incremental updates** | Full graph rebuild needed when docs change — dealbreaker for dynamic content |
| **Infrastructure** | Separate database, sync complexity, new failure modes |
| **Neo4j CE licensing** | Restricts SaaS platforms deriving primary value from Neo4j |
| **Scale justification** | Graph engines justify at >100K nodes. Business KBs are typically <10K |
| **Multi-hop queries** | Graph wins here (91% vs 43%), but these are <20% of business KB queries |

**If entity/relationship queries become needed later**: PostgreSQL recursive CTEs or Apache AGE (openCypher in PostgreSQL) — zero new infrastructure.

### What Actually Moves the Needle

| Technique | Effort | Expected Gain | Source |
|-----------|--------|---------------|--------|
| Contextual Retrieval (Anthropic) | Low | **49-67% fewer retrieval failures** | Anthropic 2024 |
| Hybrid search (BM25 + vector) | Low | 1-9% recall improvement | Industry consensus |
| Cross-encoder reranking | Medium | Significant precision boost | kapa.ai, Glean |
| Better chunking (semantic) | Low | Highest variance reduction | Production teams |

These four techniques, stacked, will dramatically outperform adding Neo4j — at a fraction of the effort and cost.

## Architecture: Enhanced Pipeline

```
Current pipeline:
  Query → Embed → pgvector search → Top-K chunks → LLM → Answer

Enhanced pipeline:
  Query
    │
    ├─► 1. Contextual embedding (query stays same)
    │
    ├─► 2a. Vector search (pgvector, dense embeddings)
    ├─► 2b. Full-text search (tsvector, BM25-style)
    │
    ├─► 3. Reciprocal Rank Fusion (merge 2a + 2b)
    │
    ├─► 4. Cross-encoder reranking (top 50 → top 10)
    │
    ├─► 5. Feedback score blending (#35)
    │
    └─► 6. LLM generation with enriched context
```

## 1. Contextual Retrieval (Anthropic's Technique)

**The single highest-ROI improvement.** Prepend a short contextual summary to each chunk before embedding, so the embedding captures document-level context — not just the chunk in isolation.

### How It Works

At indexing time (when document is chunked):
```
Original chunk:
  "Standard delivery to EU member states takes 4-6 weeks."

Contextual chunk (stored + embedded):
  "From logistics-policy-v3.pdf, Section 2.1 European Union Delivery:
   Standard delivery to EU member states takes 4-6 weeks."
```

The contextual prefix is generated by an LLM call at indexing time:
```
System: "Given the full document and this chunk, write a 1-2 sentence
         context prefix that situates this chunk within the document.
         Be concise. Include the document name and section."

Input: {full_document_text} + {chunk_text}
Output: "From logistics-policy-v3.pdf, Section 2.1 European Union Delivery:"
```

### Why This Works
- Chunks like "The rate is 15%" become "From pricing-policy.pdf, Premium Tier pricing: The rate is 15%"
- Vector search now matches on context, not just isolated fragment
- **49% fewer retrieval failures** (67% with reranking added)

### Cost
- ~$1 per million document tokens at indexing time (one-time per document)
- Slightly larger chunks stored (context prefix adds ~20-50 tokens)

### Implementation
- Add `contextual_content` column to `kb_chunks` (text, stores prefix + content)
- Generate context on chunk creation using Claude API (haiku for cost efficiency)
- Embed `contextual_content` instead of raw `content`
- Return raw `content` in search results (user doesn't see the prefix)

## 2. Hybrid Search: Vector + Full-Text

Add PostgreSQL full-text search (`tsvector`) alongside pgvector. This catches keyword-exact matches that semantic search misses (acronyms, product codes, proper nouns).

### Implementation

```sql
-- Add tsvector column to kb_chunks
ALTER TABLE kb_chunks ADD COLUMN search_vector tsvector;

-- Populate from content
UPDATE kb_chunks SET search_vector = to_tsvector('english', content);

-- GIN index for fast full-text search
CREATE INDEX idx_kb_chunks_search ON kb_chunks USING gin(search_vector);

-- Trigger to auto-update on insert/update
CREATE TRIGGER kb_chunks_search_update
  BEFORE INSERT OR UPDATE ON kb_chunks
  FOR EACH ROW EXECUTE FUNCTION
  tsvector_update_trigger(search_vector, 'pg_catalog.english', content);
```

### Query

Run both searches in parallel:
```sql
-- Vector search (existing)
SELECT id, content, 1 - (embedding <=> query_embedding) AS vector_score
FROM kb_chunks WHERE account_id = ? AND knowledge_base = ?
ORDER BY embedding <=> query_embedding LIMIT 50;

-- Full-text search (new)
SELECT id, content, ts_rank(search_vector, plainto_tsquery('english', ?)) AS text_score
FROM kb_chunks WHERE account_id = ? AND knowledge_base = ?
  AND search_vector @@ plainto_tsquery('english', ?)
ORDER BY text_score DESC LIMIT 50;
```

### Why Both?
- Vector search: "What are the shipping options?" → finds semantically related chunks
- Full-text search: "GDPR Article 17" → exact keyword match that vector search may rank lower
- Combined: catches both semantic and lexical queries

## 3. Reciprocal Rank Fusion (RRF)

Merge results from vector and full-text search into a single ranked list.

```typescript
function reciprocalRankFusion(
  vectorResults: ScoredChunk[],
  textResults: ScoredChunk[],
  k: number = 60 // smoothing constant
): ScoredChunk[] {
  const scores = new Map<string, number>();

  vectorResults.forEach((chunk, rank) => {
    scores.set(chunk.id, (scores.get(chunk.id) || 0) + 1 / (k + rank + 1));
  });

  textResults.forEach((chunk, rank) => {
    scores.set(chunk.id, (scores.get(chunk.id) || 0) + 1 / (k + rank + 1));
  });

  // Sort by fused score descending
  return [...scores.entries()]
    .sort((a, b) => b[1] - a[1])
    .map(([id, score]) => ({ id, rrf_score: score }));
}
```

Simple, parameter-free, well-proven in production (Elasticsearch, Azure AI Search).

## 4. Cross-Encoder Reranking

After RRF produces ~50 candidates, rerank with a cross-encoder for precision.

### Options

| Reranker | Cost | Latency | Quality |
|----------|------|---------|---------|
| Cohere Rerank | $0.002/1K queries | ~200ms | Excellent |
| Jina Reranker v2 | $0.001/1K queries | ~150ms | Very good |
| Claude (inline) | Token cost | ~500ms | Good, but expensive |
| Local model (bge-reranker) | Free | ~100ms | Good, self-hosted |

### Implementation

```typescript
async function rerankChunks(
  query: string,
  chunks: Chunk[],
  topK: number = 10
): Promise<Chunk[]> {
  // Option A: Cohere Rerank API
  const response = await cohere.rerank({
    query,
    documents: chunks.map(c => c.content),
    top_n: topK,
    model: 'rerank-v3.5'
  });

  return response.results.map(r => ({
    ...chunks[r.index],
    rerank_score: r.relevance_score
  }));
}
```

### Why Rerank?
- Bi-encoder (embedding) search is fast but approximate — it encodes query and chunk separately
- Cross-encoder scores query-chunk pairs **together** — much more accurate but slower
- Standard pattern: fast search (100+ candidates) → slow rerank (top 10)

## 5. Semantic Chunking (Upgraded)

Current chunking: fixed-size (~512 tokens) with 10% overlap. Upgrade to structure-aware + semantic boundaries.

### Improvements

**a. Page-level metadata preservation**
- Already extract page numbers — ensure chunk boundaries prefer page breaks
- Never split mid-sentence

**b. Section-aware splitting**
- Detect headings (PDF: font size, DOCX: heading styles, MD: `#`)
- Start new chunk at section boundaries
- Include section heading in every chunk from that section

**c. Variable chunk sizes**
- Short sections (<256 tokens): keep as single chunk (don't pad)
- Long sections: split at paragraph boundaries within section
- Range: 128-768 tokens (flexible, not rigid 512)

**d. Parent-child chunks (optional, high-impact)**
- Store large "parent" chunks (1024 tokens) and small "child" chunks (256 tokens)
- Search against child chunks (more precise matching)
- Return parent chunk to LLM (more context)
- Two-column approach: `content` (parent) + `search_content` (child), embed child

## 6. Future: Lightweight Graph (If Needed)

If entity/relationship queries become >20% of traffic, add graph **within PostgreSQL**:

### Option A: Recursive CTEs (zero extensions)
```sql
-- Entity relationships stored in a simple table
CREATE TABLE kb_entities (
  id uuid, knowledge_base uuid, name text, type text, -- person, org, policy, product
  chunk_id uuid -- which chunk mentioned this entity
);

CREATE TABLE kb_entity_relations (
  id uuid, source_entity uuid, target_entity uuid,
  relation_type text, -- 'mentions', 'supersedes', 'requires', 'contradicts'
  chunk_id uuid
);

-- Multi-hop traversal
WITH RECURSIVE entity_graph AS (
  SELECT target_entity, 1 AS depth
  FROM kb_entity_relations WHERE source_entity = ?
  UNION ALL
  SELECT r.target_entity, eg.depth + 1
  FROM kb_entity_relations r JOIN entity_graph eg ON r.source_entity = eg.target_entity
  WHERE eg.depth < 3
)
SELECT * FROM entity_graph;
```

### Option B: Apache AGE (openCypher in PostgreSQL)
```sql
-- Full Cypher syntax, runs inside PostgreSQL
SELECT * FROM cypher('kb_graph', $$
  MATCH (a:Entity {name: 'GDPR'})-[:REQUIRES]->(b:Entity)
  RETURN b.name, b.type
$$) AS (name text, type text);
```

**Decision**: defer until entity queries are a proven need. Vector + full-text + reranking solves 90%+ of use cases.
