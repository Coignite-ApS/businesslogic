Networking & Security

  • Hetzner Private Network (10.0.0.0/16) — all inter-server traffic stays private, no public internet exposure for DB/Redis
  • Firewall rules: S2 accepts connections only from S1 and S3 private IPs on ports 5432/6379. S3 exposes only 3000 (Formula API) and 3100 (Flow Trigger) via Traefik on S1
  • Cloudflare — DNS, CDN, WAF, DDoS protection. Full (strict) SSL mode. API routes bypass cache
  • TLS — Traefik on S1 handles Let's Encrypt certificates for all domains. Internal traffic unencrypted (private network, trusted)
  • Coolify SSH — S1 manages S2 and S3 via SSH keys (no password auth)

5. Execution Engine

5.1 DAG with Controlled Back-Edges

Structural DAG with annotated loop edges. Each back-edge requires a guard condition + max iteration limit (default: 100). Same concept as your formula engine's Gauss-Seidel cycles, but at flow level. Implemented in flow-engine/src/dag/mod.rs with petgraph DiGraph, forward/back edge classification, and separate topo sort on forward-only subgraph.

5.2 Execution Algorithm

1. Worker pulls job: XREADGROUP GROUP workers consumer-{id} COUNT 1 BLOCK 5000
   STREAMS flow:execute:critical flow:execute:normal flow:execute:batch
2. Load flow definition (local cache → PG fallback)
3. Initialize ExecutionContext { $trigger, $env, $meta, $nodes: {}, $last: null }
4. Build petgraph DAG, topological sort (forward edges only)
5. Phase 1: Walk topo order sequentially. Phase 2+: Ready queue with Tokio JoinSet.
6. For each node:
   a. Check budget limit ($meta.cumulative_cost vs flow.budget_limit_usd)
   b. Build NodeInput { config, data: $last, context_snapshot }
   c. Execute node handler via NodeRegistry
   d. On success: set_node_output() → update $nodes + $last, track cost
   e. On failure: apply ErrorStrategy (Retry/Fallback/Skip/Abort)
   f. Phase 2+: Check downstream deps, add ready nodes to queue
   g. Back-edges: if guard=true AND iterations < max → re-queue loop body
7. On success: XACK, persist summary to bl_flow_executions, PUBLISH flow:events:{flow_id}
8. On failure: per-node retry (exponential backoff) → fallback branch → abort → dead-letter

8. Formula Engine Integration

Three Integration Modes

Inline Formula Expressions

Any node output mapping can use Excel formulas: "=IF($input.country=\"DK\",$input.total*0.25,0)"

Calls formula_core::eval_single() — microsecond overhead.

Calculator Node

Reference existing Directus-managed calculator by ID. Reuses your Calculator struct with I/O mappings. Direct Rust call, no HTTP.

Sheet Processing Node

Full eval_sheet() — multi-sheet, cross-references, dependency graph, cycle resolution. Your entire formula engine inside one flow node.

String Interning Strategy

Make Workbook generic over interner type. formula-napi uses ThreadedRodeo (backward compat). Flow engine uses Rodeo (single-threaded, ~20% faster, per-execution isolation).

Observability Stack

  • Structured logging: tracing crate with tracing-subscriber (env-filter + JSON). Already wired in both trigger and worker main.rs
  • Distributed tracing: OpenTelemetry spans per flow execution, per node. Trace ID propagated through execution context. Dependencies in Cargo.toml (opentelemetry + tracing-opentelemetry)
  • Metrics: Prometheus-compatible endpoint on each worker (same as your /memory-stats/metrics pattern). Key metrics: flows_executed_total, flow_duration_seconds, node_duration_seconds, queue_depth, active_flows
  • Health endpoint: GET /health on trigger (built) and worker (Phase 2). Returns instance capacity, queue depth, active flows
  • Sentry integration: via sentry-tracing crate. Automatic error capture with execution context attached. Same DSN as your Directus Sentry
