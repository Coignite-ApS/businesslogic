3. BUDGET & COST ENFORCEMENT (5-LAYER ARCHITECTURE)
================================================================================

Layer 1: Per-Flow Budget
  • Field: flows.ai_budget_cents (PostgreSQL)
  • Check: check_budget() pre-validates accumulated cost
  • Enforcement: Abort node if exceeded

Layer 2: Daily Per-Account Budget
  • Storage: Redis key format redis://account:{account_id}:daily_ai_cost
  • TTL: 25 hours (INCRBYFLOAT, automatic expiry)
  • Check: compare against flows.daily_budget_cents per account

Layer 3: Monthly Per-Account Budget
  • Storage: PostgreSQL bl_account_budgets table
  • Columns: account_id, month (YYYY-MM), spent_cents
  • Logic: UPDATE ... SET spent_cents = spent_cents + $1 WHERE account_id = $2
  • Enforcement: Hard cap per month (configurable per account)

Layer 4: Global Daily Budget
  • Storage: Redis key AI_GLOBAL_DAILY_LIMIT_USD (env var)
  • Granularity: All accounts combined
  • Fallback: If Redis unavailable, returns Ok() (graceful degradation)

Layer 5: Flow-Level Budget Circuit Breaker
  • Node execution checks current cost vs. budget
  • Error handling: Retry (with backoff), Fallback (alternate branch), Skip, Abort

Storage Mechanisms:
  • Pre-check: record_cost(model, input_tokens, output_tokens) → calculate_cost()
  • Post-record: UPDATE bl_account_budgets after execution completes
  • Cancellation propagates cost to budget immediately

Cost Calculation:
  • Model pricing table in provider.rs (opus/sonnet/haiku)
  • Formula: (input_tokens * model.input_price_per_m + output_tokens * model.output_price_per_m) / 1_000_000
  • Stored in PostgreSQL bl_executions.ai_cost_cents for audit trail

Opt-In Behavior:
  • All budget layers return Ok() if PostgreSQL or Redis unavailable (resilience first)
  • Admin can override per-flow via flows.skip_budget_check boolean
  • Monthly budget enforced at account level; daily budget per-account and global

4. EXECUTION ENGINE & DAG ALGORITHM
================================================================================

Main Algorithm (execute_flow in executor/mod.rs):
  1. Load flow definition from PostgreSQL
  2. Build DAG via petgraph::DiGraph::from_edges()
  3. Validate: Check for invalid back-edges (only loop nodes allowed)
  4. Topological sort: All-node ordering respecting dependencies
  5. Ready queue: Initially contains nodes with zero dependencies
  6. Main loop:
     - Spawn ready nodes via JoinSet::spawn_blocking() or tokio::spawn()
     - Await completion with JoinSet::join_next()
     - Update dependencies for downstream nodes
     - Handle back-edges: queue back to front of DAG if guard condition passes
     - Apply error strategy (Retry, Fallback, Skip, Abort)
     - Checkpoint state at configured intervals

Execution Context Evolution:
  • Starts with: $trigger (webhook payload), $env (secrets), $meta (execution ID, account)
  • Accumulates: $nodes.{node_id}.output after each node
  • $last: Most recent node output (simplified reference)
  • Passed to each node via ExecutionContext struct
  • Interpolation: ${nodes.fetch_data.output} → resolved in node handler

Parallel Execution Model:
  • JoinSet allows multiple independent nodes to run concurrently
  • Dependencies enforced: cannot execute node until all inputs ready
  • Cancellation: CancellationToken propagated to all spawned tasks
  • Timeout: Per-flow max_execution_time (default 300s)

Error Strategies (per-node config):
  • Retry: Backoff (exponential, max 3 attempts), then Abort if all fail
  • Fallback: Jump to alternate branch on error
  • Skip: Log error, continue to next node
  • Abort: Terminate entire flow immediately

Back-Edge Handling (Loops):
  • loop node wraps around with: guard condition + max_iterations limit
  • Prevents infinite loops; maximum 1000 iterations per loop (configurable)
  • Full ExecutionContext available within loop; outputs accumulated

State Checkpointing:
  • Snapshots ExecutionContext periodically (every 100ms or after critical node)
  • Stored in Redis with key format: flow:state:exec-{execution_id}
  • On worker crash: Next available worker claims message and resumes from checkpoint
  • Deleted on successful completion

5. WORKER POOL & HORIZONTAL SCALING
================================================================================

Redis Streams Consumer Group Model:
  • 3 priority streams: critical, normal, batch
  • Consumer group: "workers" (created by trigger and worker startup)
  • Per-worker: XREADGROUP BLOCK 1000 COUNT 10 (10-msg batch, 1s block timeout)
  • Acknowledgment: XACK stream group {message_id} after successful execution
  • Claim recovery: XCLAIM every 60s for orphaned messages (worker death)

Job Lifecycle:
  1. Trigger service (flow-trigger) receives webhook/cron/manual trigger
  2. Validates: flow permissions, flow definition exists, account limits
  3. Produces: Message to Redis Stream (format: FlowExecution serialized)
  4. Stream: /critical/{account_id}, /normal/{account_id}, or /batch/{account_id}
  5. Worker: XREADGROUP consumes message, deserializes to FlowExecution struct
  6. Execution: Worker calls executor.execute_flow() (Tokio multi-threaded)
  7. Result: Stored in PostgreSQL bl_flow_executions, streamed to client if SSE enabled
  8. Acknowledge: XACK removes message from stream

Graceful Shutdown:
  • Signal handler: SIGTERM → cancel_token.cancel()
  • In-flight executions: Wait up to grace_period_secs (default 30s) for completion
  • Acknowledgment: Messages NOT acknowledged roll back to stream
  • Reconnect: Next worker picks up unacknowledged messages

Horizontal Scaling Strategy:
  • Add worker processes: No coordination needed (consumer group auto-distributes)
  • Consumer group: Stateless; each worker maintains: in-memory JoinSet + Redis connection pool
  • Throughput: 50 executions/min per worker (empirically tuned on CPX31)
  • Bottleneck: PostgreSQL connection pool (max 20 connections; tune via DATABASE_MAX_CONNECTIONS env var)

Health Push:
  • Every 30s: worker pushes health status to Redis key worker:{worker_id}:health
  • Trigger service: Monitors health keys; raises alert if stale (>90s)
  • Orchestration: Coolify detects dead worker, restarts container

Metrics:
  • Arc<AtomicU64> messages_processed per worker
  • Arc<AtomicU64> active_executions (incremented on spawn, decremented on completion)
  • Exposed via /health endpoint: {"workers": [...], "throughput_5m": N}

7. MULTI-TENANT ISOLATION & SECURITY
================================================================================

Account Scoping (CRITICAL):
  • Every query includes: AND account_id = $1 (PostgreSQL)
  • Every Redis key includes: account:{account_id}:* (Redis)
  • Every vector search includes: AND account_id = $2 (pgvector queries)

Examples:
  -- Query knowledge base chunks
  SELECT * FROM bl_kb_chunks WHERE id = $1 AND account_id = $2;

  -- Budget check
  SELECT spent_cents FROM bl_account_budgets
  WHERE account_id = $1 AND month = $2;

Authentication:
  • Webhook: HMAC-SHA256 signature verification (X-Signature header)
  • Admin: X-Admin-Token header (constant-time comparison)
  • Service-to-service: Placeholder for mTLS (Phase 7)

Authorization (RequiredRole system):
  • Core nodes: no additional role needed
  • Database node: requires "admin" role per account
  • Redis node: requires "admin" role per account
  • LLM node: requires "ai" role per account (optional per-account feature flag)
  • Validated in node handler via node.required_role validation

WASM/QuickJS Sandboxing:
  • WASM nodes: Wasmtime runtime (memory-isolated, no host access except via imports)
  • QuickJS nodes: Sandboxed global scope (no eval, no require, limited built-ins)
  • Both: Cannot access external systems without explicit node handler bridge

Data Protection:
  • API keys: Never logged; loaded from $env at runtime
  • Webhooks: Body logged only if DEBUG_LOG_PAYLOADS=true
  • Execution output: Tiered storage (inline JSON, Redis references, S3 streaming)

  • account_id (UUID), month (VARCHAR: YYYY-MM)

  • id (UUID), account_id (UUID), name (TEXT)

bl_kb_chunks (Embedded Documents):

  • created_at (TIMESTAMP)
  • INDEX: (kb_id, account_id) for fast retrieval
  • INDEX: (account_id) for multi-tenant isolation

bl_node_types (Node Registry):
  • id (VARCHAR: unique node type name)
  • category (VARCHAR: core|wasm|external)
  • schema (JSONB: input/output definitions)
  • required_role (VARCHAR: nullable, e.g., "admin", "ai")

Serialization:
  • JSONB storage for flow definitions and execution contexts

15. KEY TECHNICAL INSIGHTS & ARCHITECTURE PATTERNS
================================================================================

1. DAG Execution with JoinSet
   → Enables true parallelism without callback hell
   → Dependencies tracked via petgraph; ready queue ensures ordering
   → Cancellation token propagates cleanup across all tasks

2. Redis Streams Consumer Groups
   → Horizontal scaling without central orchestrator
   → XCLAIM recovery handles worker failures automatically
   → Account-scoped streams prevent cross-tenant job leakage

3. Five-Layer Budget Architecture
   → Resilient: Each layer independent; system degrades gracefully
   → Auditable: Costs stored in multiple places (PostgreSQL for audit, Redis for real-time)
   → Flexible: Per-flow, per-account, global limits all enforceable

4. ExecutionContext Accumulation Pattern
   → $nodes.*.output and $last enable rich data flow
   → Tiered storage (inline/Redis/S3) handles payloads of any size
   → Checkpoint recovery allows resumption after worker crash

5. Account Scoping as Default
   → Every SQL query includes account_id filter (no exceptions)
   → Vector search always scoped (security by design)
   → Budget tracking per-account (prevents cross-tenant cost leakage)

6. Error Strategy Isolation
   → Per-node configuration (not global)
   → Enables graceful degradation (Fallback) vs. hard fail (Abort)
   → Distinguishes transient (Retry) from permanent errors (Skip + Fallback)

7. WASM + QuickJS Sandboxing
   → Plugins isolated from main process
   → No access to host unless explicit node handler imports
   → Enables community contributions without security risk

8. Hybrid Storage Model
   → Inline: Fast JSON serialization for <64KB
   → Redis: Reference pointers for 64KB–100MB
   → S3: Streaming (zero-copy) for >100MB payloads
