# BusinessLogic Design Guide + Directus Theme Extension

**Date:** 2026-04-22
**Status:** Approved design, ready for implementation planning
**Scope:** Single-project spec (no decomposition needed)

## Goal

Extract BusinessLogic's visual identity from the public marketing site (https://www.businesslogic.online/) into a versioned design guide, then package that identity as a Directus theme extension selectable at `/admin/settings/appearance`.

After this lands:

- `designguide/` exists as the canonical source of truth for BusinessLogic visual tokens (consumed by humans via HTML, by code via JSON).
- Admins can switch `/admin/settings/appearance` to "BusinessLogic Light" or "BusinessLogic Dark" and the CMS visually matches the marketing site.

## Non-Goals

- No custom Vue component overrides inside Directus. Tokens only, plus `rules.custom` CSS where tokens can't reach.
- No custom login-page module.
- No replacement of the Directus logo in `project_logo`/`project_color` (those live in `directus_settings`, not in a theme extension — separate follow-up).
- No redesign of existing BusinessLogic extensions (ai-assistant, calculators, etc.).
- No font changes if the marketing site uses system fonts.

## Deliverables

Two artifacts, built in order:

1. **`designguide/`** (repo root) — the design system. Built first.
2. **`services/cms/extensions/local/project-extension-theme/`** — the Directus bundle extension. Generated from `designguide/tokens.json`.

### Deliverable 1: `designguide/`

```
designguide/
├── README.md                  # How to read, update, and consume this guide
├── index.html                 # Standalone interactive preview (no build step)
├── tokens.md                  # Canonical human-readable tokens table
├── tokens.json                # Machine-readable tokens consumed by the theme extension
├── tokens.raw.json            # Raw extraction evidence (audit trail, timestamped)
├── typography.md              # Font stack, scale, pairings, usage examples
├── color.md                   # Palette rationale, contrast pairs, dark-mode derivation rules
├── components.md              # Button, input, card, nav patterns composed from tokens
├── inspiration.md             # Reference sources + one-line takeaways per source
└── images/
    ├── source/                # Screenshots captured from businesslogic.online
    ├── brand/                 # Logo, wordmark, favicon exports
    └── previews/              # Screenshots of index.html showing components in both themes
```

**`index.html` constraints:** self-contained, inline CSS populated from `tokens.json` at author time. No bundler, no npm install. Opens in any browser offline.

**`tokens.json` schema** (single source feeding the extension):
```json
{
  "version": "1.0.0",
  "source": "https://www.businesslogic.online/",
  "extractedAt": "2026-04-22T...",
  "light": {
    "color": { "background": "#...", "surface": "#...", "primary": "#...", ... },
    "typography": { "fontFamily": { "sans": "...", "mono": "..." }, "scale": { ... } },
    "space": { "1": "4px", "2": "8px", ... },
    "radius": { "sm": "...", "md": "...", "lg": "..." },
    "shadow": { "sm": "...", "md": "...", "lg": "..." }
  },
  "dark": { ... same shape as light ... }
}
```

### Deliverable 2: `services/cms/extensions/local/project-extension-theme/`

```
project-extension-theme/
├── package.json               # Directus bundle manifest (type: "bundle")
├── src/
│   ├── index.ts               # Bundle entry registering both theme entries
│   ├── businesslogic-light.ts # Light theme: maps tokens.json → Directus theme schema
│   ├── businesslogic-dark.ts  # Dark theme
│   └── tokens.ts              # Generated from designguide/tokens.json (prebuild copy)
└── __tests__/
    └── tokens-map.test.ts     # Asserts every tokens.json field has a Directus rules.* mapping
```

**Directus theme schema** (fixed by Directus 11 — see https://directus.io/docs/guides/extensions/theme):
Theme extensions export `{ id, name, appearance, rules }`. `rules` maps to Directus's theme token tree (`background`, `foreground`, `foregrounds.normal`, `primary`, `secondary`, `borderColor`, `borderRadius`, `borderWidth`, `navigation.*`, `sidebar.*`, `form.*`, `banner.*`, `popover.*`, `custom`).

**Build integration:**
- Add Makefile target `ext-theme` mirroring existing patterns (`ext-calculators`, `ext-account`, etc.).
- Add `ext-theme` to the `ext` aggregate target and extension list in project CLAUDE.md.
- Uses `@directus/extensions-sdk` like every other local extension.
- `src/tokens.ts` is generated by a `prebuild` npm script that reads `designguide/tokens.json` and emits a typed TypeScript module. Never edited by hand. `tokens.json` is the single source of truth.

**Distribution:** Once built and CMS restarted, the two themes appear in `/admin/settings/appearance` → Theme dropdown. Users select per-appearance (light/dark). No seeding step required — Directus auto-discovers the extension.

## Extraction Workflow

Chrome DevTools MCP sequence against `https://www.businesslogic.online/`:

1. **Navigate** to the homepage.
2. **Take snapshot** — full DOM + computed styles baseline.
3. **Screenshot** hero, feature sections, footer, nav, any `<button>` states on hover (via `evaluate_script` to trigger `:hover`-equivalent via JS, or screenshot at rest and document hover as derived).
4. **Extract via `evaluate_script`:**
   - `document.documentElement.style.cssText` + computed styles of `:root` for CSS custom properties
   - `getComputedStyle()` on h1/h2/h3/body/a/button/nav/main to capture canonical type + color usage
   - `document.fonts` → list all font families loaded
   - `<link rel="stylesheet">` URLs (for @font-face discovery) — fetch relevant CSS and parse
   - `<img>` src list for logo + brand mark identification
5. **Download logo assets** — SVG if present, else PNG at highest resolution available, to `designguide/images/brand/`.
6. **Write `tokens.raw.json`** with everything extracted, ISO-8601 timestamp. This file is the audit trail — never edited by hand.
7. **Curate `tokens.json`** — Claude selects and names tokens from the raw extraction, applying design judgment: naming `brand-primary` vs. `blue-600`, picking the canonical type scale, deciding what's a real token vs. one-off. The curated `tokens.json` is reviewed in the generated `designguide/index.html` preview before the theme extension is built — that preview is the approval gate.
8. **Derive dark theme** if the marketing site is light-only: invert surfaces while preserving brand hue, run WCAG AA contrast checks on every foreground/background pair, document derivation rules in `color.md`.
9. **Capture inspiration screenshots** from Linear, Vercel dashboard, Stripe dashboard (public marketing + public product screenshots), save to `designguide/images/source/inspiration/`, write takeaways to `inspiration.md`. Not copied — used only to inform how extracted tokens get composed into components.

## Token → Directus Mapping Strategy

The theme extension is mostly a mapping layer. The table below is semantic — **exact Directus `rules.*` key names must be verified against Directus 11's actual theme schema during implementation** (the schema is the authoritative source; the table shows intent):

| BusinessLogic token         | Directus `rules.*` target                          |
|-----------------------------|----------------------------------------------------|
| `color.background`          | `background`                                       |
| `color.surface`             | `navigation.background`, `sidebar.background`      |
| `color.primary`             | `primary`                                          |
| `color.text`                | `foreground`, `foregrounds.normal`                 |
| `color.textMuted`           | `foregrounds.subdued`                              |
| `color.border`              | `borderColor`                                      |
| `color.accent`              | `secondary`                                        |
| `radius.md`                 | `borderRadius`                                     |
| `typography.fontFamily.sans`| `fontFamily` (+ `rules.custom` for `--theme--fonts--sans--font-family`) |
| `typography.fontFamily.mono`| `fontFamilyMonospace`                              |

Gaps (tokens with no direct Directus target) land in `rules.custom` as raw CSS. The test suite fails if `tokens.json` has a field that isn't mapped OR a mapping points at a non-existent `tokens.json` key.

## Testing

- **`tokens-map.test.ts`** — for every leaf in `tokens.json.light`, assert the corresponding Directus mapping exists and points at a known rules path. Same for dark. Also assert there are no orphan mappings (rules path referenced but token missing).
- **Visual sanity check** — after `make ext-theme && make cms-restart`, open `/admin/settings/appearance`, select BusinessLogic Light, screenshot. Compare manually against `designguide/index.html` light preview. Same for dark. This step is a manual QA pass, documented in the implementation plan — not automated.
- **No contract tests** — theme extension has no HTTP surface.
- **No DB migration** — theme extensions auto-discover; no `directus_settings` changes required for users to opt in.

## Defaults

- Both light and dark themes ship. Dark derived from light if marketing site is light-only.
- Tokens only in the theme (no Vue overrides), with `rules.custom` escape hatch when Directus tokens are insufficient.
- If marketing site uses a custom web font, self-host in the extension (not CDN) so admin works offline or behind strict CSPs. Directus CSP already allows `'self'` fonts.
- System fonts as fallback in every font stack.

## Open Questions

None — defaults above are explicit. User can override any of them during implementation if extraction surfaces surprises.

## Success Criteria

1. `designguide/` exists, committed, `index.html` opens standalone and renders both light and dark previews with all component examples.
2. `services/cms/extensions/local/project-extension-theme/` builds via `make ext-theme` without errors.
3. After `make cms-restart`, `/admin/settings/appearance` exposes "BusinessLogic Light" and "BusinessLogic Dark" as selectable themes.
4. Selecting either theme applies tokens visible across the admin UI (sidebar, nav, primary buttons, form inputs, content surfaces).
5. `tokens-map.test.ts` passes; every `tokens.json` key has a Directus mapping.
6. `tokens.raw.json` is present as audit trail with ISO timestamp.
