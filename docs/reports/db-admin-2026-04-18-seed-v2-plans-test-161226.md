## Phase 2 — Research Findings

- `public.subscription_plans` was empty (0 rows) before apply.
- Discovered TWO bugs in the generator-emitted SQL that would have caused `apply` to fail:
  1. **Bad ON CONFLICT clause:** `subscription_plans_unique_published` is a partial UNIQUE INDEX (mig 004), NOT a CONSTRAINT. `ON CONFLICT ON CONSTRAINT <name>` syntax fails with `constraint "..." does not exist`. Fixed by switching to index-inference syntax: `ON CONFLICT (module, tier) WHERE status = 'published'`.
  2. **Missing id default:** `subscription_plans.id` (uuid) has no DEFAULT. INSERTs without `id` fail with `null value in column "id"`. Fixed by adding `id = gen_random_uuid()` to every INSERT.
- Both fixes verified via dry-run (BEGIN/INSERT/ROLLBACK).
- Enums `module_kind` (calculators, kb, flows) and `tier_level` (starter, growth, scale, enterprise) match all 12 rows exactly.

## Risk note — Stripe test catalog drift

This migration is environment-specific (Stripe TEST product/price IDs). If the Stripe test workspace is reset or `create-products-v2.ts` is re-run with different IDs, this migration becomes stale. The UPSERTs in this migration would then re-insert with the OLD IDs (still present), but new subscriptions created against fresh Stripe IDs wouldn't match.

**Workaround when this happens:**
1. Re-run `services/cms/extensions/local/project-extension-stripe/scripts/create-products-v2.ts`
2. Regenerate `v2-catalog-output.sql`
3. Author a new migration `0NN_seed_v2_plans_test_<refresh>.sql` that UPSERTs the new IDs (existing rows update via the `(module, tier)` partial unique index).
4. Apply via `/db-admin`.
