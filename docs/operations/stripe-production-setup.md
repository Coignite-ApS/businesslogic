# Stripe Production Setup — Pricing v2 Runbook

**For:** First-time deployment of the v2 pricing catalog to production Stripe + production database.
**Prerequisites:** Stripe live mode account, production Postgres, prod env vars set.
**Estimated time:** 30–45 minutes including verification.
**Reversibility:** Mostly reversible (Stripe products can be archived; SQL can be rolled back via paired `_down.sql`). Webhook deliveries are not reversible — if you misconfigure the endpoint, real customer charges may fail until corrected.

---

## 0. Mental model

Production deployment is **two parallel tracks** that converge at the webhook step:

```
  TRACK A: Stripe                    TRACK B: Database
  ────────────────                   ───────────────────
  1. Get sk_live_ key                3. Apply schema migrations
  2. Run catalog script              4. Apply seed migration
  ─────────┬───────                  ─────────┬─────────
           │                                  │
           └──────── (5) Connect ─────────────┘
                          │
                          ▼
                  6. Configure webhook
                          │
                          ▼
                  7. Smoke test
```

**Hard rule:** never run the catalog script against production without first verifying the test-mode run worked end-to-end.

## 2. Track A — Create the Stripe live catalog

### Step 2.1: Verify your live key

```bash
# Sanity check — should print "sk_live_..." (DO NOT log the full key)
echo "${STRIPE_SECRET_KEY:0:8}..."
```

If you see `sk_test_` → STOP. You're pointed at test mode. Set the correct env var.

### Step 2.2: Run the catalog script

From the Stripe extension directory:

```bash
cd services/cms/extensions/local/project-extension-stripe
STRIPE_SECRET_KEY=sk_live_... npm run stripe:create-v2-products
```

The script will:
- Detect live mode (`sk_live_` prefix)
- Print `Mode: 🔴 LIVE` + a 5-second abort window
- Iterate 23 products: 12 base subscriptions + 3 recurring add-ons + 5 one-time add-ons + 3 wallet top-ups
- For each: search by `metadata.product_key` first (idempotent), create or update accordingly
- For each base subscription: create 4 prices (EUR monthly + EUR annual + USD monthly + USD annual)
- For one-time products: create 2 prices (EUR + USD)
- Output `scripts/v2-catalog-output.sql` with production product/price IDs

### Step 2.3: Verify in Stripe Dashboard

- Go to: https://dashboard.stripe.com/products?metadata%5Bpricing_version%5D=v2
- Confirm 23 products listed
- Spot-check Calculators Growth: should have 4 active prices (EUR €79/mo, EUR €790/yr, USD $89/mo, USD $890/yr)

If anything looks wrong:
- Re-run the script (idempotent — will update only what changed)
- If a product is wrong, archive it manually in Dashboard, then re-run

## 3. Track B — Apply the seed migration

### Step 3.1: Move the generated SQL into a migration

The script wrote `scripts/v2-catalog-output.sql`. Make it a permanent migration:

```bash
# Find next available migration number
ls migrations/cms/0*.sql | sort | tail -3
# Suppose latest is 018_seed_v2_plans_test.sql → use 019 for production seed

cp services/cms/extensions/local/project-extension-stripe/scripts/v2-catalog-output.sql \
   migrations/cms/019_seed_v2_plans_prod.sql

# Add a header comment marking it production-only
sed -i '' '1i\
-- PRODUCTION SEED — apply ONLY against the production database.\
-- Generated by scripts/create-products-v2.ts run with sk_live_... key.\
-- Test-mode equivalent is migrations/cms/018_seed_v2_plans_test.sql (do not apply both to same DB).\
' migrations/cms/019_seed_v2_plans_prod.sql
```

Write the paired down migration:

```bash
cat > migrations/cms/019_seed_v2_plans_prod_down.sql <<'EOF'
-- Down migration for production v2 plan seed.
-- Removes only the rows inserted by 019_seed_v2_plans_prod.sql, identified by
-- their stripe_product_id values.

DELETE FROM public.subscription_plans
WHERE stripe_product_id IN (
  -- Paste the prod_xxx IDs from 019_seed_v2_plans_prod.sql here.
  -- Run: grep stripe_product_id migrations/cms/019_seed_v2_plans_prod.sql | grep -oE "prod_[A-Za-z0-9]+"
);
EOF
```

Fill in the `prod_xxx` IDs in the down file from the `019_*.sql` file. (Future improvement: have the generator emit the down SQL too.)

### Step 3.2: Apply via /db-admin

From a Claude Code session pointed at production:

```
/db-admin seed-v2-plans-prod
```

The agent will:
1. Take pre-snapshot
2. Verify the migration file structure (idempotent ON CONFLICT, valid SQL)
3. Run `make diff` → should show 12 INSERTs
4. Phase 4.5 audit: confirm `subscription_plans` is empty (or warn if rows already exist)
5. Apply via `psql -1`
6. Phase 6.5 verify: 12 rows present, all linked to live Stripe IDs
7. Write dated report

If `subscription_plans` already has rows from a prior pricing version → STOP. Don't apply. Instead, see §10.

## 4. Convergence: verify Stripe ↔ DB linkage

After both tracks complete:

-- All 12 plans should have non-null Stripe IDs
SELECT module, tier, name, stripe_product_id, stripe_price_monthly_id, stripe_price_annual_id

- `https://dashboard.stripe.com/prices/<stripe_price_monthly_id>` → confirms the price is active

## 5. Step 6 — Configure Stripe webhook

### Step 5.1: Create the webhook endpoint in Stripe

1. Stripe Dashboard → Developers → Webhooks → **+ Add endpoint**
2. **Endpoint URL:** `https://<your-prod-domain>/stripe/webhook`
3. **Events to send** (select these specific events):
   - `checkout.session.completed`
   - `customer.subscription.created`
   - `customer.subscription.updated`
   - `customer.subscription.deleted`
   - `invoice.payment_succeeded`
   - `invoice.payment_failed`
   - `payment_intent.succeeded`
   - `product.updated`
4. **API version:** match the project's pinned version (`2024-12-18.acacia` per `services/cms/extensions/local/project-extension-stripe/src/stripe-client.ts`)
5. Click **Add endpoint**
6. Stripe shows a **signing secret** (`whsec_...`)

### Step 5.2: Set the webhook secret in production

```bash
# Update the production environment:
# infrastructure/docker/.env (or equivalent secret manager)
STRIPE_WEBHOOK_SECRET=whsec_<the-value-Stripe-just-showed-you>

# Restart the CMS service to pick up the new env var
make cms-restart
# OR if using Coolify / systemd: trigger your normal restart
```

### Step 5.3: Verify webhook delivery

In the Stripe Dashboard webhook details page, click **Send test webhook**:
- Pick `payment_intent.succeeded`
- Click **Send test webhook**
- Wait ~5 seconds
- Refresh the page → "Recent events" should show the test event with status `Succeeded`

If status is `Failed`:
- Check the response code in Stripe Dashboard
- Most common issues:
  - `401 Invalid signature` → `STRIPE_WEBHOOK_SECRET` is wrong or service didn't pick up the env update
  - `404 Not Found` → URL typo or service not reachable from internet
  - `500 Server Error` → check application logs

## 6. Step 7 — Smoke test

### Step 6.1: Create a test account in production

(In production but with a real human's account — there's no "test mode" inside live Stripe.)

1. Sign up via the production registration page
2. Verify in DB: `SELECT * FROM public.account WHERE name = '<test-name>';`
3. Verify wallet auto-credit: `SELECT balance_eur FROM public.ai_wallet WHERE account_id = '<id>';` → should be `5.0000`
4. Verify ledger entry: `SELECT * FROM public.ai_wallet_ledger WHERE account_id = '<id>';` → should be one credit row, source='promo'

### Step 6.2: Test a real subscription

⚠️ **This will charge a real card.** Use your own card or a colleague's with a low-cost tier (Calculators Starter = €19/mo).

1. From the test account, navigate to `/admin/account/subscription`
2. Click **Activate** on Calculators
3. Choose Starter / Monthly
4. Complete Stripe Checkout with your real card
5. Watch for webhook delivery in Stripe Dashboard → status `Succeeded`
6. Verify in DB: `SELECT * FROM public.subscriptions WHERE account_id = '<id>';` → should have one row with `module='calculators'`, `tier='starter'`, `status='trialing'` (14-day trial), `stripe_subscription_id='sub_...'`
7. Verify in account UI: subscription card shows "Calculators Starter — Trialing — Trial ends YYYY-MM-DD"

### Step 6.3: Cancel + refund

Once you've verified the flow works:
1. Stripe Dashboard → Customers → find your test customer → Subscriptions → cancel immediately
2. Refund the trial setup charge (if any)
3. Webhook should deliver `customer.subscription.deleted` → DB row updates `status='canceled'`
4. Account UI updates to show "Not active — Activate"

### Step 6.4: Test the wallet top-up flow

1. From the test account, click **Top up €20** on the AI Wallet card
2. Complete Stripe Checkout with your real card
3. Watch for webhook delivery (`payment_intent.succeeded`) → status `Succeeded`
4. Verify in DB:
   - `SELECT balance_eur FROM ai_wallet WHERE account_id = '<id>';` → should be `25.0000` (5 signup + 20 top-up)
   - `SELECT * FROM ai_wallet_topup WHERE account_id = '<id>';` → should have one row with `amount_eur=20`, `expires_at` ~12 months out
   - `SELECT * FROM ai_wallet_ledger WHERE account_id = '<id>' ORDER BY occurred_at DESC LIMIT 3;` → top-up credit + signup credit
5. Refund via Stripe Dashboard

## 7. Operational reference

### Webhook event flow

```
Stripe event delivered
        │
        ▼
POST /stripe/webhook (Express handler in project-extension-stripe)
        │
        ├─ Verify signature against STRIPE_WEBHOOK_SECRET
        │  (rejects if HMAC mismatch — protects against forged events)
        │
        ├─ Parse event
        │
        ├─ Route by event.type:
        │     checkout.session.completed     → handleCheckoutCompleted
        │     customer.subscription.created  → handleSubscriptionUpdated (treats create+update same)
        │     customer.subscription.updated  → handleSubscriptionUpdated
        │     customer.subscription.deleted  → handleSubscriptionDeleted
        │     invoice.payment_succeeded      → handleInvoicePaymentSucceeded
        │     invoice.payment_failed         → handleInvoicePaymentFailed
        │     payment_intent.succeeded       → handlePaymentIntentSucceeded (wallet top-up)
        │     product.updated                → handleProductUpdated (sync price changes)
        │
        └─ Each handler wrapped in withIdempotency(event.id)
           (skips if event already processed)
```

### Catalog updates after launch

If you change a price or feature allowance:

1. Edit `BASE_PRODUCTS` / `RECURRING_ADDONS` / etc. in `scripts/create-products-v2.ts`
2. Re-run the script with `sk_live_` key
3. For changed prices: old price is archived (`active=false`), new price created
4. **Existing customer subscriptions keep paying the OLD price** — Stripe doesn't retroactively migrate
5. To migrate existing customers to new prices: use Stripe Dashboard or write a one-shot migration script using `stripe.subscriptions.update(id, { items: [{ id: item_id, price: new_price_id }] })`

### Re-syncing after Stripe Dashboard changes

If someone makes manual changes in the Stripe Dashboard (e.g., updating product description):
- Re-run the catalog script — it will detect drift and update the DB
- Or wait for the next `product.updated` webhook to fire `handleProductUpdated`

## 8. Rollback procedures

### Webhook misfiring

If webhook handlers are corrupting data:
1. **Disable the endpoint** in Stripe Dashboard (Webhooks → click endpoint → Disable)
2. Stripe will retry undelivered events for up to 3 days; events queue up
3. Fix the handler bug, deploy
4. Re-enable the endpoint — Stripe replays queued events
5. The `withIdempotency()` wrapper prevents duplicate processing

### Bad seed migration

If `019_seed_v2_plans_prod.sql` produces wrong rows:
1. Run the down migration: `psql -f migrations/cms/019_seed_v2_plans_prod_down.sql`
2. Fix the source SQL or regenerate via the script
3. Re-apply

### Catastrophic schema corruption

Restore from pre-task PG dump:
```bash
gunzip -c infrastructure/db-snapshots/pre_seed-v2-plans-prod_<ts>.sql.gz | \
  psql -U directus -d directus
```

### Stripe account compromise

If you suspect the live key has leaked:
1. Stripe Dashboard → Developers → API keys → Roll the live secret key
2. Update `STRIPE_SECRET_KEY` in production env immediately
3. Restart CMS service
4. Audit recent activity in Stripe Dashboard logs for suspicious calls

## 9. Monitoring after launch

### What to watch in the first 48 hours

| Metric | Where | Healthy? |
|---|---|---|
| Webhook success rate | Stripe Dashboard → Webhooks → endpoint stats | > 99% |
| New signups → wallet credit | `SELECT COUNT(*) FROM ai_wallet_ledger WHERE source='promo' AND occurred_at >= NOW() - INTERVAL '48h'` | should match new account count |
| Subscriptions created | `SELECT module, tier, COUNT(*) FROM subscriptions WHERE date_created >= NOW() - INTERVAL '48h' GROUP BY module, tier` | match Stripe's reported new subs |
| Trial conversions | (manual review at end of trial period) | — |
| AI calls returning 402 | Application logs | should be 0 unless wallet legitimately empty |
| Stripe webhook deliveries failing | Stripe Dashboard | investigate every failure |

### Alerts to set up (via your monitoring system)

- Webhook failure rate > 1% over 5 minutes → page oncall
- `ai_wallet.balance_eur = 0` for active accounts (potential UX problem) → daily report
- Pending payment_intents > 1 hour old → manual review (likely failed checkout)
- Production DB size growth (`ai_wallet_ledger`, `usage_events`) — set baseline + alert at 2× growth rate

## 10. Migrating from a prior pricing version

If production already has v1 (or v1.5) `subscription_plans` rows:

1. **Don't** run `019_seed_v2_plans_prod.sql` directly — it would create duplicate rows or hit the unique constraint
2. Instead, write a custom migration:
   - Archive existing v1 plans: `UPDATE subscription_plans SET status='archived' WHERE created_at < '2026-04-18'` (or similar marker)
   - Add the new v2 rows from the script's SQL output
   - Migrate any active customer subscriptions to v2 plan IDs (one-shot script using Stripe `subscriptions.update`)
3. Test the migration on a snapshot first (`make snapshot SLUG=pre-v2-cohort-migration` → restore on a temp DB → dry-run)
4. Coordinate with customers — communicate any pricing changes ahead of time

This scenario doesn't apply to the current codebase (no live customer); kept here for future reference.

If the setup fails midway and you need to abandon:

```bash
# 1. Archive Stripe products (don't delete — they're referenced by historical records)
#    In Stripe Dashboard, filter by metadata.pricing_version=v2
#    Click each → Archive

# 2. Remove DB rows
psql -f migrations/cms/019_seed_v2_plans_prod_down.sql

# 3. Disable the webhook endpoint in Stripe Dashboard

# 4. Restore env vars to their prior values

# 5. Restart services
```
